## 1. 文档目的与范围

**目的**:为"基于 `open-multi-agent`(TS 编排骨架)+ `ai_uitest_agent`(领域参考)+ 历史用例 `测试用例.xlsx`"构建 Web UI 自动化测试平台,提供实施级技术设计基线,供团队对齐后进入编码。

**范围**:
- 涵盖架构、模块边界、关键算法、数据模型、API 集成、Agent 定义、安全、分阶段计划。
- 所有引用 `@open-multi-agent/core` 的签名均来自源码实测(标注 `文件:行号`),可直接照此编码。
- 不含逐行代码实现(P0 落地时再细化),但包含接口契约与关键伪代码。

**非范围**:移动端、接口/API 测试、性能测试。当前用例全部为 `pc后台` 单端口 PC Web,平台聚焦此场景。

## 2. 背景、目标与关键决策

### 2.1 三个输入

| 输入 | 角色 | 关键事实 |
|------|------|---------|
| `ai_uitest_agent/`(Python) | **领域参考**(非代码直搬) | "AI"是假的(用例生成=规则模板、执行靠猜动作、选择器兜底返回裸 `"div"`);LangGraph 装饰性(decide 是硬编码状态机、tool_node 死节点、未接 API);多处阻断 Bug。可平移的是**设计**:DB schema、Vue 七页面 UI、Skills 目录规范、WebSocket 进度消息类型。 |
| `open-multi-agent/`(TypeScript,`@open-multi-agent/core` v1.9.0) | **编排骨架**(直接依赖) | Coordinator 模式目标→任务 DAG;`TaskQueue` 拓扑调度+级联失败+独立任务继续;`executeWithRetry` 指数退避;Checkpoint/Restore;13 个 LLM provider 惰性加载;`defineTool`+Zod 工具框架;`SharedMemory` 命名空间 KV;`delegate_to_agent` 委派;3 个运行时依赖。 |
| `测试用例.xlsx`(776KB) | **核心数据** | 3566 条纯自然语言中文用例;零元素定位、零形式化断言;5 级树 `项目→端口→模块→功能→子功能→用例`;Sheet2 是 365 行菜单分类树;**样式损坏**(openpyxl/pandas 崩,需 XML 直读);三作者风格差异大(汝霖 780 条无标点流水句)。 |

### 2.2 三项关键决策(已锁定)

1. **TS 优先**:以 `open-multi-agent` 为编排骨架,Web 测试能力作为 `tools`/`agents` 嫁接;Vue UI 保留改指向 Node 后端。不在 Python 里重造编排框架。
2. **历史用例回归为主**:3566 条 NL 用例自动跑通是首要目标;NL 解释器 + 元素定位是核心;AI 探索生成(`runTeam`)二期。
3. **导入时一次性结构化**:导入时 LLM 把每条用例转成结构化 `steps`+`assertions` 存库;执行确定、可人工编辑、成本一次性;~20 动词词典做高覆盖兜底,LLM 处理剩余。运行时不重复调 LLM 翻译。

### 2.3 核心设计张力与取舍

回归测试需要**确定性 + 稳定 + 低成本**,而 NL 用例天然**模糊**。解法:双模式执行器——
- **确定性执行器(DeterministicCaseRunner)**:对结构化良好的用例,直接按 `TestStep[]` 顺序调用浏览器工具,**无 LLM 在环**,最快最稳。
- **Agent 兜底**:仅当步骤歧义、定位失败、断言需判断时,`delegate` 给 interactor/assertor agent。

这兼顾了"回归稳定"与"NL 容错",且把 LLM 成本压到最小。

## 4. 技术栈选型

| 层 | 选型 | 理由 |
|----|------|------|
| 语言/运行时 | TypeScript(ESM)+ Node ≥ 20 | open-multi-agent 原生;Playwright TS 原生;strict 类型安全 |
| 编排框架 | `@open-multi-agent/core` | 直接依赖,不 fork;DAG/retry/checkpoint/13 provider/工具框架全有 |
| 浏览器自动化 | Playwright(TS) | 原生 TS、选择器引擎强(role/text/css/xpath)、auto-wait、trace viewer |
| LLM | 经 open-multi-agent adapter(默认 anthropic/openai;支持 OpenAI 兼容 baseURL 接入私有/国产模型) | 复用其抽象,不自己造 |
| 后端框架 | Fastify | 高性能、TS 友好、插件体系;轻量替代:Hono |
| ORM | Drizzle ORM | 类型安全、SQL 透明、迁移清晰;替代:Kysely |
| 数据库 | SQLite(开发)/ Postgres(生产) | 与 ai_uitest_agent 一致起步,平滑升级 |
| 前端 | Vue 3 + Element Plus + Pinia + ECharts | 取自 ai_uitest_agent,改指向 Node 后端 |
| 实时 | WebSocket(ws) | 转发编排进度 |
| 校验 | Zod | 与 open-multi-agent 一致(defineTool/​outputSchema 都用 Zod) |
| 测试 | Vitest | 与 open-multi-agent 一致;单测+集成 |
| 进程管理 | 单 Node 进程 + 浏览器上下文池 | 回归场景并发可控 |

**依赖克制**:除 `@open-multi-agent/core` 三个运行时依赖外,新增控制在 playwright、fastify、drizzle、ws、zod(已传递)、vitest(dev)。不引入 apscheduler/weasyprint/jinja2 等 ai_uitest_agent 里的冗余依赖。

## 5. 与 open-multi-agent 的集成(精确 API)

> 以下签名均来自源码实测。导入方式:`import { OpenMultiAgent, Team, defineTool, ... } from '@open-multi-agent/core'`(主入口 `src/index.ts:57-234`)。

### 5.1 实例化编排器

`OpenMultiAgent` 构造:`constructor(config: OrchestratorConfig = {})`(`orchestrator.ts:1630`)。

```ts
import { OpenMultiAgent, FileStore } from '@open-multi-agent/core'

const orchestrator = new OpenMultiAgent({
  defaultModel: process.env.OMA_MODEL ?? 'claude-sonnet-4-6',
  defaultProvider: (process.env.OMA_PROVIDER as SupportedProvider) ?? 'anthropic',
  defaultApiKey: process.env.OMA_API_KEY,
  defaultBaseURL: process.env.OMA_BASE_URL,        // OpenAI 兼容私有/国产模型
  maxConcurrency: Number(process.env.BROWSER_POOL_SIZE ?? 3), // 与浏览器实例数对齐
  defaultToolPreset: 'readonly',                   // 默认最小授权,按 agent 显式授予
  checkpoint: { store: new DbMemoryStore(db) },    // 落库,见 §10.4
  onProgress: (e) => wsBus.broadcast(runId, toWsMessage(e)),  // → WebSocket,见 §12
  onTrace:    (e) => wsBus.broadcast(runId, toWsTrace(e)),
  onAgentStream: (agent, ev) => wsBus.broadcast(runId, toWsStream(agent, ev)),
})
```

`OrchestratorConfig` 关键字段(`types.ts:1064-1152`):`maxConcurrency`(默认 5)、`maxDelegationDepth`(默认 3)、`maxTokenBudget`、`defaultModel`/`defaultProvider`/`defaultBaseURL`/`defaultApiKey`、`checkpoint: boolean | CheckpointOptions`、`defaultToolPreset`、`defaultCwd`、`onProgress`/`onTrace`/`onApproval`/`onPlanReady`/`onAgentStream`。

### 5.2 Team 与 Agent 定义

`Team` 是类,`TeamConfig`(`types.ts:714-728`):`{ name, agents: AgentConfig[], sharedMemory?, sharedMemoryStore?, maxConcurrency? }`。

```ts
const team = orchestrator.createTeam('regression-team', {
  name: 'regression-team',
  agents: [interactor, assertor, pageExplorer, debugger],  // 见 §13
  sharedMemory: true,
  sharedMemoryStore: new DbMemoryStore(db),   // 命名空间 KV 落库

```

`AgentConfig`(`types.ts:377-628`)关键字段:`name`、`model?`、`provider?`/`baseURL?`/`apiKey?`、`systemPrompt?`、`customTools?: ToolDefinition[]`(注册即授权)、`tools?: string[]`(内置工具白名单)、`toolPreset?`、`cwd?`、`maxTurns?`、`maxTokenBudget?`、`contextStrategy?`、`outputSchema?: ZodSchema`(强制结构化输出)、`loopDetection?`、`beforeRun?`/`afterRun?`。

### 5.3 三个执行入口

```ts
// 单 agent 一次性(用于导入时 NL 解释、单页探索)
runAgent(config: AgentConfig, prompt: string, options?: { abortSignal? }): Promise<AgentRunResult>

// 目标 → 任务 DAG(coordinator 自动分解)— 二期 AI 探索
runTeam(team: Team, goal: string, options?: RunTeamOptions): Promise<TeamRunResult>

// 显式任务管线(用户定义 dependsOn)— 回归主路径
runTasks(team: Team, tasks: ReadonlyArray<RunTaskSpec>, options?: RunTasksOptions): Promise<TeamRunResult>
```

**回归主路径用 `runTasks`**。`RunTaskSpec`(`types.ts:772-784`):

```ts
interface RunTaskSpec {
  title: string
  description: string
  assignee?: string                 // agent 名;省略则 Scheduler 自动分配
  dependsOn?: string[]              // 引用其他 task 的 **title**(内部解析为 id)
  memoryScope?: 'dependencies' | 'all'
  maxRetries?: number               // 默认 0;UI 测试建议 1-2
  retryDelayMs?: number             // 默认 1000
  retryBackoff?: number             // 默认 2
  role?: string
  priority?: 'low' | 'normal' | 'high' | 'critical'
  verify?: ConsensusVerifyOptions
}
```

`dependsOn` 用 **title 字符串**引用(实测 `examples/basics/task-pipeline.ts:152`)。

`AgentRunResult`(`types.ts:683-707`):`success`、`output`、`messages`、`tokenUsage`、`toolCalls`、`structured?`(outputSchema 验证后的对象)、`loopDetected?`、`budgetExceeded?`、`error?`。

### 5.4 工具定义

`defineTool`(`tool/framework.ts:71-111`):

```ts
function defineTool<TInput>(config: {
  name: string
  description: string
  inputSchema: ZodSchema<TInput>
  outputSchema?: ZodSchema<string>     // 验证 ToolResult.data
  maxOutputChars?: number
  execute: (input: TInput, context: ToolUseContext) => Promise<ToolResult>
}): ToolDefinition<TInput>
```

`ToolResult`(`types.ts:296-300`):`{ data: string, isError?: boolean, metadata?: { tokenUsage? } }`。**注意 `data` 恒为 string**(结构化数据由调用方 JSON 序列化)。

`ToolUseContext`(`types.ts:214-243`):`{ agent: {name,role,model}, team?, abortSignal?, abortController?, cwd?, metadata? }`。工具内可通过 `context.abortSignal` 响应停止,通过 `context.metadata` 读取 runId/caseId 等注入上下文。

真实最小示例(`src/index.ts:39-50`):

```ts
import { z } from 'zod'
const myTool = defineTool({
  name: 'fetch_data',
  description: 'Fetch JSON data from a URL.',
  inputSchema: z.object({ url: z.string().url() }),
  execute: async ({ url }) => {
    const res = await fetch(url)
    return { data: await res.text() }
  },
})
```

**关键不变量**(来自 `CLAUDE.md`):内置工具 `bash`/`file_*`/`grep`/`glob`/`delegate_to_agent` 是**默认拒绝**——必须经 `tools` 或 `toolPreset` 显式授权;`customTools` 注册即授权。文件工具在 `cwd` 沙箱内,`bash` 不沙箱。**工具错误不抛异常**,返回 `ToolResult(isError: true)`;任务失败级联到依赖者,独立任务继续。

### 5.5 进度回调(→ WebSocket 的接入点)

框架通过 `OrchestratorConfig` 的**三个回调**暴露进度(无 EventEmitter/AsyncIterable,均为函数回调):

| 回调 | 类型 | 用途 |
|------|------|------|
| `onProgress` | `(e: OrchestratorEvent) => void` | 任务/agent 生命周期(粗粒度) |
| `onTrace` | `(e: TraceEvent) => void \| Promise<void>` | 细粒度 span(LLM 调用、工具调用、token) |
| `onAgentStream` | `(agent, e: StreamEvent) => void` | 实时文本/推理增量 |

`OrchestratorEvent`(`types.ts:1047-1061`):

```ts
interface OrchestratorEvent {
  type: 'agent_start' | 'agent_complete' | 'task_start' | 'task_complete'
      | 'task_skipped' | 'task_retry' | 'budget_exceeded' | 'message' | 'error'
  agent?: string
  task?: string
  data?: unknown
}
```

`TraceEvent`(`types.ts:1335-1436`):联合类型,含 `llm_call`/`tool_call`/`task`/`agent`/`plan_ready`/`agent_stream`/`consensus`,公共字段 `runId`/`spanId`/`parentId`/`type`/`startMs`/`endMs`/`durationMs`/`agent`/`taskId?`。`tool_call` 含 `tool`/`isError`/`input`/`output`(已脱敏)。

详见 §12 的消息映射表。

### 5.6 MemoryStore 与 Checkpoint 接入

`MemoryStore`(`types.ts:1479-1497`):

```ts
interface MemoryStore {
  get(key: string): Promise<MemoryEntry | null>
  set(key: string, value: string, metadata?: Record<string, unknown>): Promise<void>
  setWithExpiry?(key, value, expiresAtTurn, metadata?): Promise<void>  // 可选
  list(): Promise<MemoryEntry[]>
  delete(key: string): Promise<void>
  clear(): Promise<void>
}
```

`value` 恒为 string。注入方式:共享内存经 `TeamConfig.sharedMemoryStore`;checkpoint 经 `OrchestratorConfig.checkpoint.store` 或 `RunTasksOptions.checkpoint.store`(省略时默认用团队的共享内存 store,`orchestrator.ts:2873-2882`)。

`SharedMemory`(`memory/shared.ts:67-529`)公开:`write(agentName, key, value, metadata?, options?)`、`writeExpiring(agentName, key, value, ttlTurns, ...)`、`read(key)`、`listAll()`、`listByAgent(name)`、`getSummary(filter?)`、`advanceTurn()`、`snapshot()`/`restore()`、`getStore()`。**键命名空间 `<agentName>/<key>`**。

`CheckpointOptions`(`types.ts:1159-1171`):`{ enabled?, store?, key?, runId? }`。每完成一个 task 自动存一次(`orchestrator.ts:1144`)。`restore(team, tasks|plan|options, options?)` 三重载(`orchestrator.ts:2179`):无 checkpoint 则回退正常运行,有则重建队列、跳过已完成、续跑剩余。

`PlanArtifact`/`runFromPlan(team, plan, options?)`(`orchestrator.ts:2142`):可把一次 `runTeam` 的 plan 序列化为可重放 artifact,不重跑 coordinator——**回归套件固化用**:首次 `runTeam` 生成 plan → `createPlanArtifact` 存库 → 后续 `runFromPlan` 直接执行,保证回归可重复。

### 5.7 LLM 配置

13 个 provider(`adapter.ts:42`):`anthropic | azure-openai | bedrock | copilot | deepseek | doubao | grok | hunyuan | minimax | mimo | openai | gemini | qiniu`。`createAdapter(provider, apiKey?, baseURL?, region?)` 惰性加载;用户不直接调,框架内部按 `AgentConfig.provider`/`apiKey`/`baseURL` 调用,缺省回退环境变量(如 `ANTHROPIC_API_KEY`/`OPENAI_API_KEY`)。

**OpenAI 兼容私有/国产模型**(实测 `examples/providers/openrouter.ts`):`provider: 'openai'` + `baseURL` + 非空 `apiKey` 占位即可对接 Ollama/vLLM/国产网关。建议平台默认支持配置私有网关。

### 5.8 delegate_to_agent(失败自愈/调试)

`delegate_to_agent`(`tool/built-in/delegate.ts:24-109`)仅在 `runTeam`/`runTasks` 池 worker 中注册。参数 `{ target_agent, prompt }`。限制:禁自委派、须在名册、禁循环(`chain.includes(target)`)、深度 ≤ `maxDelegationDepth`(默认 3)、池槽位 ≥1。被委派 agent 经 `pool.runEphemeral` 在**全新会话**运行,token 计入父预算。

**用法**:interactor 执行失败 → `delegate_to_agent({ target_agent: 'debugger', prompt: '步骤 X 点击【保存】失败,快照已存,分析原因并给出修复后的定位' })`。

### 5.9 集成约束清单

1. **不 fork 源码**:仅通过 `OrchestratorConfig`/`AgentConfig`/`defineTool`/`CoordinatorConfig`/`MemoryStore` 扩展点定制,保留升级能力。
2. **`data` 恒 string**:工具返回结构化数据需 `JSON.stringify`。
3. **内置工具默认拒绝**:浏览器工具全部走 `customTools`(注册即授权),不依赖内置 `bash`/`file_*`。
4. **`asyncio` 陷阱不存在**:TS 原生 async,无 ai_uitest_agent 的 `@tool` 内 `asyncio.run` 问题。
5. **进度是回调不是流**:WS 转发需在回调里 `wsBus.broadcast`(回调同步调用,`onTrace` 异步返回会被吞 rejection,故内部不抛)。

## 6. 浏览器工具层

基于 `defineTool` + Playwright TS。每个工具 `inputSchema` 用 Zod 校验,`execute` 返回 `{ data: string }`(结构化结果 JSON 序列化)。

### 6.1 统一元素定位 schema

所有交互工具接受统一的 `Locator` 输入(详见 §8):

```ts
const LocatorSchema = z.object({
  type: z.enum(['alias', 'css', 'xpath', 'role', 'text', 'ref']),
  value: z.string(),                  // alias 名 / css / xpath / text / ref id
  role: z.string().optional(),        // type=role 时
  name: z.string().optional(),        // type=role 时可访问名
}).describe('元素定位:alias 优先解析别名词典,ref 用快照编号,ai 兜底')
```

### 6.2 工具清单

| 工具 | 输入 | 输出(data) | 说明 |
|------|------|-----------|------|
| `browser_navigate` | `{ url, waitUntil? }` | `{ url, title }` | 导航并 auto-wait |
| `browser_snapshot` | `{}` | accessibility tree + `ref` 编号 | 供 LLM 探索/定位;ref 可复用 |
| `browser_click` | `{ locator }` | `{ clicked: true }` | 定位→点击;失败返回 isError |
| `browser_type` | `{ locator, text, clear? }` | `{ filled: true }` | clear 默认 true |
| `browser_select` | `{ locator, value }` | `{ selected: true }` | 下拉选择 |
| `browser_check` | `{ locator, checked }` | `{ checked }` | checkbox/radio |
| `browser_get_text` | `{ locator }` | `{ text }` | 读取文本(断言用) |
| `browser_assert` | `{ kind, locator?, expected?, timeout? }` | `{ pass: boolean, actual }` | 见 §6.3 |
| `browser_screenshot` | `{ fullPage?, locator? }` | `{ path }` | 落 `screenshots/`,返回相对路径 |
| `browser_locate` | `{ description }` | `{ ref, locator }` | AI 语义定位兜底(§8) |
| `browser_wait` | `{ condition, timeout? }` | `{ ok: true }` | wait_for_load_state/wait_for_selector |
| `browser_get_dom` | `{ locator? }` | `{ html }` | 受限 HTML 片段(调试/定位辅助) |

### 6.3 断言工具 `browser_assert`

`kind` 枚举:`'text'`(元素文本含/等)|`'visible'`|`'hidden'`|`'url'`(当前 URL 匹配)|`'title'`|`'count'`(匹配元素数)|`'value'`(输入值)|`'enabled'`|`'checked'`。返回 `{ pass, actual, expected }`,由确定性执行器或 assertor agent 判定。断言失败不抛异常,返回 `pass:false`,由上层决定是否标记用例失败/重试/委派。

### 6.4 浏览器上下文池

- 每个 worker agent 对应一个独立 `BrowserContext`(隔离 cookie/localStorage),与 `maxConcurrency` 对齐。
- 回归套件中"登录"任务把登录态写入 `SharedMemory`(如 `auth/cookies`),后续任务的任务描述注入该上下文,或共享同一 context 复用登录态——优先**共享登录 context**(快),失败再隔离重试。
- headless 可配(开发用有头调试,修正 ai_uitest_agent 硬编码 headless 的缺陷)。
- `BrowserPool` 真正池化(N 个 context),修正 ai_uitest_agent "每种类型 1 实例"的缺陷。

### 6.5 选择器生成修正

ai_uitest_agent 兜底返回裸 `"div"` 无法定位——本平台**禁止仅返回 tag**。`browser_snapshot` 为每个可交互元素生成稳定 `ref`(基于 accessibility snapshot),元素定位以 `role+name`/`text`/`alias`/`ref` 优先,`css`/`xpath` 次之,从不退化为裸 tag。

## 7. NL 用例解释器

把 `测试步骤`/`预期结果` 两列自然语言,在**导入时一次性**转成结构化 `TestStep[]` + `Assertion[]`。这是回归可行性的核心。

### 7.1 流水线

```
原始用例(NL)
  │
  ├─ 7.2 步骤切分 ──────────► 句子数组(编号式 + 流水式容错)
  │
  ├─ 7.3 动作分类(词典优先) ► TestStep{action, target_desc, value, locator?}
  │        ├─ 命中词典:直接映射(无 LLM,快、确定)
  │        └─ 未命中:LLM 兜底(outputSchema 强制结构化)
  │
  ├─ 7.4 断言提取 ──────────► Assertion{kind, target, expected}
  │
  ├─ 7.5 元素别名解析 ──────► 别名登记/匹配 page-object(§8)
  │
  └─ 7.6 产出存库 ──────────► TestStep + Assertion(JSON),人工可编辑
```

### 7.2 步骤切分(规则,无 LLM)

- **编号式**(78%):正则 `/\d+[.、)]\s*/` 切分。
- **流水式**(22%,汝霖):中文无标点动词堆叠。用动词前缀词典(进入/点击/输入/选择/查看/清空/保存/确认/切换/删除/编辑/搜索/勾选/上传/下载/导出/导入/复制/拖拽/滚动)做分词切句;切不开的整句交 LLM 兜底。
- 步骤普遍短(92% ≤3 步),切分成本低。

### 7.3 动作分类(词典优先 + LLM 兜底)

**动词词典 → action 映射**(高覆盖,无 LLM):

| 中文动词/模式 | action | 备注 |
|--------------|--------|------|
| 点击/单击/按下/点【X】 | `click` | 目标=【】/""内文本 |
| 输入/填写/填入/录入 | `type` | value 来自测试数据列 |
| 选择/下拉选/切换(下拉) | `select` | |
| 勾选/取消勾选/勾选框 | `check` | |
| 进入/打开/跳转/访问 | `navigate` | 目标=页面名(需映射 URL) |
| 清空/清除 | `clear`→`type('')` | |
| 上传 | `upload` | |
| 等待/加载 | `wait` | |

词典未覆盖(约 10–15%,主要是复合句、条件句、汝霖流水句)→ **LLM 兜底**,用 `case-interpreter` agent + `outputSchema` 强制结构化输出(见 §13.1)。

### 7.4 断言提取(规则 + LLM)

`预期结果`列模式归类为 checkpoint 类型(无形式化断言,需模式抽取):

| 模式 | assertion.kind |
|------|---------------|
| "登录成功""跳转到X""进入X页面" | `url` / `title` |
| "提示XXX""弹出XXX""显示XXX" | `text`(可见文本含) |
| "按钮变灰/不可点击" | `enabled`(false) |
| "状态变为已开启/绿色" | `value` / `checked` |
| "数量减少N""新增一条记录" | `text`(数字)——需运行时取数 |
| "不允许保存""校验不通过" | `visible`(错误提示) |

复杂/含分支容忍的预期(如"弹窗关闭,或弹窗保持打开")→ LLM 兜底,产出多条可接受断言。

### 7.5 元素别名解析

步骤中的 `【登录】按钮`/`"用户名"输入框` → 提取别名 → 匹配该页面的 page-object 别名词典(§8)。匹配到则填 `locator:{type:'alias',value}`;未匹配则标记 `locator:null` + `target_desc`,执行时走 AI 兜底,并把成功定位的结果**回写别名词典**(自学习)。

### 7.6 outputSchema 强制结构化(LLM 兜底用)

`case-interpreter` agent 配 `outputSchema`(Zod),失败带错误反馈重试一次,结果取 `AgentRunResult.structured`:

```ts
const CaseInterpretationSchema = z.object({
  steps: z.array(z.object({
    action: z.enum(['navigate','click','type','select','check','clear','upload','wait','other']),
    targetDescription: z.string(),
    value: z.string().optional(),
    locator: LocatorSchema.nullable(),
    rawText: z.string(),
    confidence: z.number().min(0).max(1),
  })),
  assertions: z.array(z.object({
    kind: z.enum(['text','visible','hidden','url','title','count','value','enabled','checked']),
    target: z.string().optional(),
    expected: z.string(),
    rawText: z.string(),
    confidence: z.number().min(0).max(1),
  })),
  ambiguities: z.array(z.string()).describe('需人工确认的歧义点'),
})
```

### 7.7 导入时结构化的工程要点

- **批量并发**:3566 条分批(每批 ~20 条),并发受 `maxConcurrency` 与 LLM 速率限制约束;失败重试;进度经 `onProgress` 回推。
- **成本控制**:词典先跑(无 LLM),仅未命中走 LLM;估算 LLM 处理量约 10–15%(~400–530 条)。
- **可重入**:每条用例结构化结果落库带 `interpret_version`,重跑只处理变更或失败的。
- **人工可编辑**:结构化结果存为 JSON 字段,UI 可编辑后覆盖;`confidence < 0.7` 或 `ambiguities` 非空标记待人工确认。
- **50 条样本 gate**:导入全量前,先抽 50 条(覆盖三作者+多模块)人工校验翻译+定位准确率,达标(建议步骤动作准确率 ≥ 90%、断言可执行率 ≥ 80%)才扩量。

## 8. 元素定位层

历史用例零定位器,定位是回归稳定性的命门。

### 8.1 三级定位策略(按可靠性递减)

1. **别名词典(page object)**:为每个页面/URL 模式维护 `{ alias → locator }`。高频路径(登录、导航、常用表单)人工维护,最稳。`locator` 用 `role+name` 优先(Playwright 推荐的最稳方式),`text`/`css`/`xpath` 次之。
2. **快照 ref**:`browser_snapshot` 产出 accessibility tree + 数字 `ref`,agent/执行器用 `ref` 定位。适合 AI 探索(P6)与一次性定位。
3. **AI 语义定位兜底**:`browser_locate({ description: '【登录】按钮' })`,LLM 结合 snapshot+截图给出 locator。用于别名未命中的长尾。

### 8.2 解析流程

```
Locator{type}
  ├─ alias  ─► 查别名词典 ─► 命中:返回 locator
  │                          └ 未命中 ─┐
  ├─ ref    ─► 直接用 snapshot ref      │
  ├─ css/xpath/role/text ─► 直接用      │
  └─ ai / 未命中 ──────────────────────► browser_locate(LLM)
                                          └─ 成功 ─► 回写别名词典(自学习)
                                          └─ 失败 ─► isError,触发 delegate debugger
```

### 8.3 定位自愈

定位失败时:`browser_locate` 重试 → 仍失败 → `delegate_to_agent('debugger', ...)` 分析 snapshot+截图 → 给出修复 locator → 回写词典。同一别名连续 N 次失败标记为"待人工维护"。

### 8.4 别名词典数据来源

- **人工**:高频路径(UI 引导录入)。
- **导入时结构化**:NL 解释器提取的别名(§7.5)。
- **自学习**:AI 定位成功后回写。
- **page-explorer agent**(P6/可选):首次访问页面时自动 snapshot → LLM 提别名建议 → 人工确认入库。

## 11. 执行编排

### 11.1 回归套件 → DAG(主路径,`runTasks`)

从选中的 `test_case[]` 构建 `RunTaskSpec[]`:

```ts
function buildRegressionTasks(cases: TestCase[], ctx: RunContext): RunTaskSpec[] {
  const tasks: RunTaskSpec[] = []
  // 1. 登录前置(若套件需要且有登录用例)
  const loginCase = cases.find(c => c.module_path.includes('登录'))
  if (loginCase) {
    tasks.push(toTask(loginCase, { assignee: 'executor', priority: 'critical' }))
  }
  // 2. 其余用例,依赖登录(共享登录态)
  for (const c of cases.filter(c => c !== loginCase)) {
    tasks.push(toTask(c, {
      assignee: 'executor',
      dependsOn: loginCase ? [loginCase.title] : [],
      maxRetries: 1, retryDelayMs: 1000, retryBackoff: 2,
    }))
  }
  // 3. 报告任务,依赖全部
  tasks.push({
    title: '生成报告',
    description: `聚合 run ${ctx.runId} 全部结果生成报告`,
    assignee: 'reporter',
    dependsOn: cases.map(c => c.title),
  })
  return tasks
}
```

- 独立用例并行(受 `maxConcurrency` 与浏览器池约束);登录前置阻塞后续。
- 某用例失败 → 级联跳过**依赖它的**任务(本设计无下游用例依赖,故不影响其他独立用例,符合"独立任务继续")。
- `executor` 任务描述内嵌结构化 steps+assertions + SharedMemory 上下文(登录态/页面 URL)。

### 11.2 确定性执行器(无 LLM 在环)

`executor` agent 的 `systemPrompt` 指示:严格按任务描述中的 `steps` 顺序调用浏览器工具,执行 `assertions`,**不自由发挥**。但更稳的做法是**确定性执行器直接调工具**:

```ts
async function runDeterministic(steps: TestStep[], asserts: Assertion[], browser: BrowserCtx): Promise<CaseVerdict> {
  for (const step of steps) {
    const locator = await resolveLocator(step.locator, step.targetDescription, browser)
    const res = await dispatchAction(step, locator, browser)   // click/type/select/...
    if (res.isError) return { status: 'failed', at: step.order, error: res.data }
  }
  for (const a of asserts) {
    const r = await browser_assert({ kind: a.kind, locator: await resolveLocator(...), expected: a.expected })
    if (!r.pass) return { status: 'failed', assertion: a }
  }
  return { status: 'passed' }
}
```

**双模式**:用例标记 `execution_mode`:
- `deterministic`(默认):结构化良好(`confidence ≥ 0.7` 且无 `ambiguities`)→ 直接跑,无 LLM。
- `agent`:歧义/定位不稳 → `executor` agent 在环,失败 `delegate` debugger。

`runTasks` 的每个 case 任务,`beforeRun` 钩子按 `execution_mode` 选择路径;`executor` agent 仅在 `agent` 模式实际被调度。这样把 LLM 成本压到最小,且回归可重复。

### 11.3 重试与断点续跑

- **步骤级重试**:定位/点击失败,`browser_*` 内部 auto-wait + 1 次自愈(`browser_locate`)。
- **用例级重试**:`RunTaskSpec.maxRetries`(建议 1,UI 抖动);`executeWithRetry` 指数退避 + jitter;terminal 错误(401/403/校验错)不重试。
- **套件级续跑**:`checkpoint: { store: DbMemoryStore }`,每 case 完成存一次;中断后 `orchestrator.restore(team, tasks, { runId })` 跳过已完成、续跑剩余。
- **plan 固化**:首次跑通后 `createPlanArtifact` 存库,后续 `runFromPlan` 保证回归可重复、免重分解。

### 11.4 并发与浏览器实例对齐

`maxConcurrency` = 浏览器池大小。每个并发任务一个独立 `BrowserContext`;登录态经 SharedMemory 或共享 context 复用。`defaultToolPreset: 'readonly'` 最小授权,浏览器工具走 `customTools`。

## 12. 实时进度与可观测性

### 12.1 三回调 → WebSocket 消息映射

| 源回调 / 事件 | WS 消息 type | 前端用途 |
|--------------|-------------|---------|
| `onProgress` `task_start` | `case_start` | 用例开始,高亮 DAG 节点 |
| `onProgress` `task_complete` | `case_complete` | 用例完成(pass/fail),更新结果 |
| `onProgress` `task_retry` | `case_retry` | 重试提示 |
| `onProgress` `task_skipped` | `case_skipped` | 级联跳过 |
| `onProgress` `agent_start/complete` | `agent_lifecycle` | agent 状态 |
| `onProgress` `budget_exceeded` | `budget_exceeded` | 预算告警 |
| `onTrace` `tool_call` | `tool_call` | 工具调用流水(含已脱敏 input/output) |
| `onTrace` `llm_call` | `llm_call` | token 计数 |
| `onTrace` `task` | `task_trace` | 任务级 span(成功/重试) |
| `onAgentStream` `text`/`reasoning` | `agent_stream` | 实时文本/思考增量(agent 模式) |

回调内 `wsBus.broadcast(runId, msg)`,按 `runId` 路由到订阅该 run 的前端连接。回调同步调用,内部不抛(`onTrace` 异步 rejection 被吞)。

### 12.2 trace 落库与回放

`onTrace` 事件同时落 `oma_trace`(runId, spanId, parentId, type, agent, taskId, payload JSON, durationMs),支持运行回放与失败归因。复用 open-multi-agent `dashboard` 渲染任务 DAG(纯 HTML,无 I/O)。

### 12.3 截图与证据

每个 `run_step` 失败时自动 `browser_screenshot` + 保存 snapshot HTML;报告聚合截图链,修正 ai_uitest_agent 报告 XSS 风险(HTML 转义所有用户内容)。

## 14. 安全与合规

| 项 | 措施 |
|----|------|
| **密钥管理** | LLM API key 仅从环境变量/secret manager 读(`OMA_API_KEY` 等),不入库不入仓;`.env` git-ignore |
| **认证** | 实现 ai_uitest_agent 缺失的认证(有 User 模型无实现);API + WS 鉴权;CORS 收窄(修正 `*`) |

## 15. 分阶段交付计划

| 阶段 | 目标 | 关键交付 | 验收标准 |
|------|------|---------|---------|
| **P0 骨架** | 端到端跑通 1 条 | TS 项目 + `@open-multi-agent/core` + Playwright;最小工具集(navigate/click/type/assert/screenshot/snapshot);executor+reporter agent;`runTasks` 跑硬编码登录用例→截图→结果;WS 进度 | 一条登录用例自动执行成功并出截图,前端看板可见进度 |

| **P3 元素定位层** | 定位可靠 | page object + 别名词典;`browser_locate` AI 兜底;定位自愈 + 回写;高频路径人工录入 | 50 条样本定位成功率达标,自愈链路可用 |
| **P4 回归编排** | 套件 DAG 执行 | `buildRegressionTasks`;确定性执行器 + agent 兜底双模式;retry/checkpoint/级联失败;登录态共享;plan 固化(`runFromPlan`) | 一组用例自动跑通并出报告,中断可续跑 |

## 16. 风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| NL 翻译准确率不足,历史用例跑不通 | **高** | P2 50 条样本 gate;词典优先压 LLM 量;低置信标记人工;双模式(executor 兜底) |

| open-multi-agent 升级破坏集成 | 中 | 不 fork,仅用扩展点;锁版本;集成测试覆盖关键路径 |
